# IT 详细设计 Agent 开发计划 v1（基于框架 v3）

## 1. 目标与策略

### 1.1 目标
在最短周期内建立“可执行、可校验、可被前端/后端/测试/SRE直接消费”的项目级详细设计体系，并逐步实现 Agent 自动化生成与校验。

### 1.2 核心策略（交付物先行）
1. 先定义交付物标准，再开发能力。
2. 先打磨模板与 Schema，再接入 Agent 生成。
3. 先通过门禁跑通“人工+半自动”流程，再全自动化。

---

## 2. 总体里程碑（6 周）

| 阶段 | 周期 | 核心目标 | 阶段验收（Exit Criteria） |
|---|---|---|---|
| M0 启动与基线冻结 | 第 1 周前 2 天 | 锁定 v3 范围、角色诉求、项目边界 | `范围清单`、`角色诉求清单`、`版本基线` 完成并评审通过 |
| M1 交付物标准化 | 第 1-2 周 | 完成交付物目录、字段规范、模板、命名规范 | MUST 交付物模板与 Schema 完整，评审通过 |
| M2 校验与门禁 | 第 3 周 | 完成 lint/schema/追踪链校验和 CI 门禁 | CI 可阻断不合格交付，样例项目通过 |
| M3 角色消费验证 | 第 4 周 | 验证前端/后端/测试/SRE可直接使用交付物 | 四类角色消费演练通过，问题闭环 |
| M4 Agent 能力接入 | 第 5 周 | 接入 MUST 能力生成链路（先半自动） | 至少 5 类 MUST 产物可自动生成并校验 |
| M5 试点与收敛 | 第 6 周 | 在 1 个真实项目试点并收敛规范 | 试点通过，发布 v1.0 规范与模板包 |

## 3. 第一优先级：交付物标准规范与模板（M1）

### 3.1 交付物清单（MUST）
1. `openapi.yaml`
2. `errors-rfc9457.json`
3. `schema.sql`
4. `architecture.md`（Mermaid）
5. `sequence-*.md`（Mermaid）
6. `state-*.md`（Mermaid）
7. `class-*.md`（Mermaid）
8. `ddd-structure.md`
9. `config-catalog.yaml`
10. `test-inputs.md`
11. `coverage-map.json`
12. `slo.yaml`
13. `observability-spec.yaml`
14. `deployment-runbook.md`
15. `traceability.json`
16. `review-checklist.md`

### 3.2 标准规范包（需先完成）
1. 命名规范：文件名、ID（`REQ-*`/`DES-*`/`TC-*`）
2. 字段规范：每个交付物必填字段、可选字段、枚举值
3. Schema 规范：JSON/YAML 的 JSON Schema
4. 图规范：Mermaid 语法与最小元素要求
5. 追踪规范：`REQ -> Design -> Test -> Runbook` 映射规则

### 3.3 模板包（需先完成）
1. 主模板：各类产物模板（含注释与示例）
2. 角色模板：前端/后端/测试/SRE 消费视图模板
3. 样例包：一个“黄金样例项目”完整设计包

### 3.4 M1 验收标准
1. MUST 交付物全部有模板。
2. JSON/YAML 交付物全部有 Schema。
3. 样例项目可通过基础校验。
4. 四类角色评审意见收敛到可执行版本。

---

## 4. 第二优先级：校验与门禁（M2）

### 4.1 校验器能力
1. OpenAPI/AsyncAPI 语法与规范校验
2. SQL 与迁移一致性校验
3. Mermaid 渲染可用性校验
4. `traceability.json` 完整性校验
5. 角色最小交付物完整性校验

### 4.2 CI 门禁策略
1. 任一 MUST 交付物缺失 -> 阻断
2. Schema 校验失败 -> 阻断
3. 追踪链断裂 -> 阻断
4. 角色交付缺失 -> 阻断

### 4.3 M2 验收标准
- 在样例项目中，CI 能准确拦截 100% 人为注入的不合规场景（缺文件、字段错、断链）。

---

## 5. 第三优先级：角色消费验证（M3）

### 5.1 角色演练脚本
1. 前端：仅用 API/错误/时序完成联调准备。
2. 后端：仅用模块/DDD/数据/集成产物完成任务拆解。
3. 测试：仅用 traceability/state/test-inputs 设计测试集。
4. SRE：仅用 SLO/观测/runbook 完成发布前检查。

### 5.2 M3 验收标准
- 四类角色均能在不补充口头信息情况下完成各自准备动作。

---

## 6. 第四优先级：Agent 能力接入（M4）

### 6.1 接入顺序（MUST 优先）
1. 架构映射
2. API 设计
3. 数据与迁移设计
4. 流程/状态设计
5. 测试可设计性
6. 可观测与运行就绪

### 6.2 接入方式
1. 先半自动：Agent 生成初稿 + 校验器门禁 + 人工确认。
2. 再自动化：收敛提示词、模板、脚本后提升自动比例。

### 6.3 M4 验收标准
- 至少 5 类 MUST 产物可自动生成并一次通过校验率 >= 80%。

---

## 7. 试点与发布（M5）

### 7.1 试点范围
- 选择 1 个中等复杂度项目（有 API、DB、流程、发布要求）。

### 7.2 试点指标
1. 设计评审轮次下降（目标 >= 20%）
2. 需求-设计-测试断链率降低（目标 <= 5%）
3. 角色满意度（前端/后端/测试/SRE）>= 4/5
4. 发布前缺陷发现提前率提升（目标 >= 30%）

### 7.3 发布产物
1. `IT详细设计Agent工作框架-v3.md`
2. `模板包 v1.0`
3. `Schema 包 v1.0`
4. `校验器与 CI 门禁规则 v1.0`
5. `试点复盘报告`

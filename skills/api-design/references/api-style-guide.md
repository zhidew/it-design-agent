# API 设计风格指南

## 目的
当 API 设计子代理生成或更新项目级 API 产物时，请遵守本指南。

## 受众规则
1. 前后端内部 API 或服务间同步 API 统一使用 `api-internal.yaml`。
2. 仅当项目明确向外部系统或合作伙伴开放 API 时，才使用 `api-public.yaml`。
3. **Internal vs Public 的区别**：
   - **api-internal.yaml**：迭代速度快，允许一定的破坏性变更，包含内部审计、详细错误堆栈等敏感信息。
   - **api-public.yaml**：版本极其稳定（如 v1, v2），强制要求向后兼容，字段经过严格脱敏，隐藏内部系统架构。
4. 当内部和公共契约在稳定性、认证或报文规则上存在差异时，必须保持它们相互独立。

## 契约规则
1. 优先使用基于名词的稳定路径，例如 `/orders` 和 `/orders/{orderId}`。
2. API 契约中的 `operationId` 应保持简短、以动词开头且全局唯一。
3. 确保请求和响应的 Schema 定义明确，避免使用未定义的自由格式对象（free-form objects）。
4. 业务错误必须符合 RFC 9457 规范字段，仅在必要时添加项目特定字段。
5. 除非需求明确授权破坏性变更，否则必须保持向后兼容性。
6. 为每个请求体和业务响应在 `components/schemas` 下定义具名 DTO。
7. 对于列表接口，必须定义分页查询参数，并在响应中包含分页元数据对象（PageMeta）。
8. 每个业务接口必须提供至少一个请求示例，以及一个成功或错误响应示例。

## 外部 API 特殊规则
1. **脱敏处理**：外部 API 严禁暴露内部数据库主键、内部工号或内部系统路径。
2. **限流提示**：必须在响应示例或描述中体现限流策略。
3. **安全性**：必须明确认证方式（如 API Key, OAuth2）。

## 证据要求
至少记录以下内容：
1. 复用或对比了哪些现有资产。
2. 契约受众是内部还是外部（或两者）。
3. 运行了哪些校验工具。

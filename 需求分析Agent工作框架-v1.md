# 需求分析 Agent 工作框架

## 1. 目标与适用范围
本框架用于将业务诉求稳定转化为可验证、可评审、可追踪的需求包，作为 IT 详细设计与研发实施的上游输入。

适用范围：
- 业务需求澄清与结构化
- FR/NFR 定义与边界控制
- 规则沉淀与影响评估
- 与 IT 详细设计 Agent 的交接

## 2. 职责边界

### 2.1 需求分析 Agent 负责
- 明确业务目标、范围、约束、优先级
- 输出功能与非功能需求（FR/NFR）
- 输出可测试验收标准（Given-When-Then）
- 输出需求追踪矩阵（至少追踪到验收项）
- 输出影响分析（业务/系统/数据/发布）

### 2.2 需求分析 Agent 不负责
- 技术选型和代码实现方案
- 详细架构设计与模块内部实现细节

## 3. 工作流（需求侧）
1. 需求澄清：目标、痛点、干系人、范围（In/Out Scope）。
2. 需求结构化：场景、规则、FR/NFR、优先级、边界条件。
3. 需求评审：冻结基线版本，定义变更入口。
4. 交接设计：将基线需求包交付 IT 详细设计 Agent。
5. 变更治理：新变更进入影响分析后再评估是否纳入版本。

## 4. 输入输出契约（I/O Contract）

### 4.1 输入
- 业务目标/KPI
- 当前流程与痛点
- 外部约束（时间、预算、合规、组织）

### 4.2 输出
- `PRD.md`（背景、范围、角色场景、业务流程）
- `requirements.json`（FR/NFR 结构化清单）
- `acceptance-tests.md`（Given-When-Then）
- `traceability-req-test.json`（`RequirementID -> AcceptanceID`）
- `impact-matrix.md`（影响分析）

## 5. 需求结构化设计维度（建议）
1. 业务目标与范围（In/Out Scope）
2. 角色与场景（主路径/例外路径）
3. 需求分解（Epic -> Feature -> Story）
4. 业务规则与状态流转
5. 输入输出契约（参数、结果、错误语义）
6. 数据需求（实体、字段、约束）
7. 异常与边界（并发、超时、重复提交）
8. 非功能需求（性能、安全、可用性、可观测性）
9. 验收标准（可测试、可度量）
10. 追踪与变更（需求版本、责任人、状态）

## 6. 规则结构化表达（标准）

### 6.1 规则卡片字段
- RuleID
- RuleName
- Domain
- Scope
- Trigger
- ConditionExpr
- Action
- ElseAction
- ExceptionHandling
- Priority
- ConflictPolicy
- Source
- Owner
- EffectiveFrom
- EffectiveTo
- Status
- TestCaseIDs

### 6.2 统一语法
```text
WHEN <触发事件>
IF <条件表达式>
THEN <执行动作>
ELSE <兜底动作>
ERROR <错误码>
```

## 7. 影响分析（必填）
每条需求至少评估：
1. 业务流程影响
2. 系统与接口影响
3. 数据模型影响
4. 非功能影响（性能/安全/可用性/监控）
5. 测试与发布影响
6. 成本与计划影响

建议模板：
```markdown
| RequirementID | ImpactObject | ImpactType | Severity | Owner | Mitigation | ETA | Status |
|---|---|---|---|---|---|---|---|
```

## 8. 质量门禁（DoD）
- 每条需求必须包含：业务价值、优先级、验收标准、边界条件。
- 需求必须有唯一编号：`REQ-*`。
- 需求必须有影响分析，不完整不得进入设计评审。
- 追踪矩阵完整度必须达到阈值（建议 >= 95%）。

## 9. 最小落地清单
1. 统一编号：`REQ-*`、`R-*`、`TC-*`。
2. 每条需求绑定至少一个验收项。
3. 评审前检查追踪矩阵与影响矩阵完整度。
4. 交接时输出基线版本号和变更日志。

## 10. 与 IT 详细设计 Agent 交接契约

### 10.1 交接输入包（必须）
- PRD 基线版本
- 需求结构化清单（FR/NFR）
- 规则与决策表
- 影响分析矩阵
- 验收标准与追踪信息

### 10.2 交接完成标准
- IT 详细设计 Agent 能直接基于输入包生成结构化设计产物
- 所有 `REQ-*` 均可在下游追踪到设计项
